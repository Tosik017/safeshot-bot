# SafeShot Bot

**Telegram-бот безопасного предпросмотра ссылок.** Берёт ссылку из группового чата, открывает её в изолированном браузере *на своей стороне*, делает скриншот и присылает в чат карточку: превью страницы, метаданные (название, цена, рейтинг) и предупреждение о безопасности. Смысл — чтобы человек **увидел** содержимое подозрительной ссылки, **не переходя** по ней сам.

> Это переработанная, security-hardened версия проекта (*hardening* — усиление защиты: целенаправленное закрытие уязвимостей и сокращение поверхности атаки). Закрыты SSRF, DNS rebinding, доступ к облачным метаданным и внутренней сети, DoS/OOM; контейнер работает не от root; конфигурация устроена по принципу fail-closed.

---

## 1. Зачем это нужно

В чатах постоянно летят ссылки: товары, новости, фишинг (*phishing* — выманивание паролей/данных карт через поддельные сайты-двойники). Перейти по незнакомой ссылке = риск: фишинг, drive-by загрузка (*drive-by* — заражение от одного лишь открытия страницы, без кликов), редирект на вредонос.

Бот разрывает этот риск: ссылку открывает **он**, в одноразовом изолированном окружении, и отдаёт людям только безопасную «фотографию» страницы плюс выжимку метаданных. Пользователю не нужно никуда переходить — он принимает решение, глядя на скриншот и плашку-предупреждение.

---

## 2. Как это работает — общая картина

Один цикл «ссылка → карточка» проходит через пять рубежей. Каждый — это и есть trust boundary (*trust boundary* — граница доверия: линия, на которой недоверенные данные проверяются перед тем, как пустить их дальше).

```
[Telegram-чат] --ссылка--> bot.py --> security.is_safe --> queue_manager --> screenshot.shoot
                           фильтры     SSRF-барьер        очередь+лимиты    Chromium (изоляция)
                              |                                                    |
                          cache.get (готовый ответ?)                       metadata.fetch (httpx)
                              |                                                    |
                              +------------------> карточка в чат <----------------+
```

1. **`bot.py`** ловит сообщение, отсеивает чужие группы, спам и дубли, достаёт URL.
2. **`security.py`** решает, безопасен ли адрес (не ведёт ли во внутреннюю сеть).
3. **`cache.py`** проверяет, не делали ли мы эту страницу недавно (если да — мгновенный ответ).
4. **`queue_manager.py`** ставит задачу в очередь под лимитами ресурсов.
5. **`screenshot.py`** рендерит страницу в Chromium и режет скриншот; параллельно **`metadata.py`** тянет метаданные. Результат собирается в карточку и кешируется.

## 3. Разбор модулей и функций

### 3.1 `config.py` — конфигурация, secure-by-default

*Secure-by-default* — «безопасно по умолчанию»: если что-то не настроено, система выбирает безопасный, а не удобный вариант.

- `BOT_TOKEN` — токен от @BotFather. Читается как обязательный: нет токена → процесс падает сразу (*fail-fast* — «падай рано»: лучше не стартовать вовсе, чем работать в сломанном состоянии).
- `ALLOWED_GROUP_IDS` — белый список (*allow-list* — перечень того, что разрешено; всё остальное запрещено) ID групп, где боту можно работать. Парсится из строки через запятую/пробел; поддержан и старый ключ `ALLOWED_GROUP_ID`.
- **Fail-closed-замок:** если список пуст **и** не задан `ALLOW_OPEN_MODE=true` — `SystemExit`, бот не стартует. *Fail-closed* — «закрываться при сбое»: при неопределённости доступ запрещается, а не открывается. Это чинит сценарий «забыл env → бот работает где угодно и становится открытым прокси».
- `DISABLED_THREADS` / `DISABLED_GENERAL_CHATS` — denylist (*denylist* — чёрный список: перечень запрещённого) топиков, где бот молчит. Формат `группа:топик`, либо `группа:general` для сообщений вне топиков.
- Тюнинг рендера и лимиты ресурсов, каждый со своим смыслом:
  - `SEMAPHORE = 1` — *semaphore* (счётчик одновременных доступов к ресурсу); единица = один скриншот за раз, потому что два Chromium = OOM на 512 МБ. *OOM* (Out Of Memory — нехватка памяти; процесс убивает ОС).
  - `TASK_TIMEOUT_SEC = 90`, `MAX_QUEUE_SIZE = 10`, `MAX_INFLIGHT_PER_CHAT = 2`, `RAM_LIMIT_MB = 430`, `RATE_LIMIT_SEC = 5`, `MAX_URL_LEN = 2048`, `MAX_BODY_BYTES = 2_000_000`.
  - `CHROMIUM_SANDBOX` — переключатель настоящего sandbox (см. §3.5).

### 3.2 `security.py` — SSRF-барьер, единственная точка проверки адреса

*SSRF* (Server-Side Request Forgery — подделка запроса со стороны сервера): атакующий заставляет твой сервер сходить туда, куда сам он не достучится, — во внутреннюю сеть, к облачным метаданным, к localhost. Для скриншот-бота это главный риск: он по своей природе ходит по чужим ссылкам.

`is_safe(url) -> bool` возвращает `True`, только если выполнено всё:

1. **Схема** (*scheme* — протокол в начале URL) ∈ {`http`, `https`}. Отсекает `file://`, `gopher://`, `ftp://`.
2. **Host не внутренний:** не `localhost` и не оканчивается на внутренние суффиксы (`.internal`, `.svc`, `.cluster.local`, `.consul`, …). Хвостовая точка снимается (`evil.com.` == `evil.com` для DNS).
3. **Порт** ∈ {80, 443}. Блокирует `:6379` (Redis), `:9200` (Elasticsearch), `:2375` (docker) даже на «публичном» IP — обычный пользователь не кидает ссылки на такие порты, а атака на внутренние сервисы кидает.
4. **Все IP-адреса хоста публичные.** Резолвинг (*resolve* — превращение доменного имени в IP) делается через `getaddrinfo`, который возвращает **и** IPv4 (A-записи), **и** IPv6 (AAAA). Проверяется *каждый* адрес; хотя бы один приватный → блок. Это закрывает dual-stack-обход (*dual-stack* — у хоста два адреса: один «чистый», другой во внутреннюю сеть).
5. **Нормализация и блок-диапазоны.** IPv4-mapped IPv6 (`::ffff:127.0.0.1`) приводится к IPv4 перед проверкой. Блокируются приватные, loopback (*loopback* — петля на саму машину, 127.0.0.0/8 и `::1`), link-local (включая `169.254.169.254` — cloud metadata, откуда крадут облачные ключи), `0.0.0.0/8`, multicast, reserved и всё, что `not is_global`.
6. **Fail-closed:** любая ошибка резолва → `False`.

Эта функция вызывается **четырежды на разных рубежах** (defense in depth — *защита вглубь*: несколько независимых барьеров, чтобы пробой одного не открывал путь): перед очередью в `bot.py`, повторно прямо перед `goto` в `screenshot.py` (сужает окно DNS rebinding), на **каждый** подзапрос внутри Chromium (route handler) и на финальный URL после редиректов в `metadata.py`.

*DNS rebinding* — атака, где домен сперва резолвится в публичный IP (проходит проверку), а через секунду — во внутренний (когда браузер реально подключается). Повторная проверка перед `goto` + проверка каждого подзапроса резко сужают это окно.

### 3.3 `bot.py` — мозг: приём сообщения и весь конвейер

Главный обработчик `handle()` идёт по шагам, отбраковывая лишнее как можно раньше:

1. **Свежесть.** Сообщения старше `MAX_MSG_AGE = 60` сек игнорируются — после перезапуска бот не разгребает протухшую историю.
2. **Фильтр группы.** Чат не в `ALLOWED_GROUP_IDS` → бот выходит из него (`leave_chat`). Хендлер `on_my_chat_member` делает то же при добавлении в чужую группу.
3. **Denylist топиков.** В отключённых топиках бот полностью молчит. Стоит *после* фильтра группы, чтобы из чужих групп всё равно выходить.
4. **Извлечение URL.** `URL_RE = https?://[^\s]+` достаёт первую ссылку. Аномально длинные (> `MAX_URL_LEN`) отбрасываются.
5. **Trusted sender.** *Доверенный отправитель* — админ группы или сообщение «от имени группы» (анонимный админ). Такие сообщения бот **пропускает целиком** (`return` до всех проверок) — он их не обслуживает, чтобы не работать инструментом в руках того, кому и так доверяет чат. Список админов кешируется на 300 сек (`_admin_cache`).
6. **Анти-спам дубликатами.** Повтор той же ссылки тем же юзером в окне `DUP_WINDOW_SEC = 120` → эскалация (*escalation* — нарастающая реакция): 1) удаление + мягкое напоминание; 2) предупреждение; 3) последнее предупреждение; 4) реальный mute на `BAN_SEC = 300` сек через `restrict_chat_member`. Счётчик strike живёт в `TTLCache` и сам затухает за `STRIKE_DECAY_SEC`. Ничего не пишется на диск — активный mute хранит сам Telegram (по `until_date`).
   - *TTLCache* (Time-To-Live cache) — кеш в памяти, где записи сами исчезают по истечении срока; `maxsize` не даёт ему расти бесконечно.
   - Все модераторские действия (`_react`, `_delete`, `_mute`, `_notice`) «мягко падают»: нет прав у бота — просто лог, без краша.
7. **Rate limit.** *Rate limiting* — ограничение частоты: между *разными* ссылками одного юзера выдерживается `RATE_LIMIT_SEC = 5` сек. Дубли сюда не доходят (их ловит шаг 6).
8. **SSRF-гейт.** `security.is_safe(url)` — небезопасный адрес → вежливый отказ.
9. **Кеш.** `cache.get(url)`: если есть готовая карточка — мгновенный ответ из кеша; если есть «негативная» запись (страница недавно не открылась) — сразу сообщение об ошибке, без повторного рендера.
10. **Очередь.** `queue_manager.enqueue` ставит задачу; при переполнении — «бот перегружен». Если та же ссылка уже в работе (*in-flight* — выполняется прямо сейчас) — бот ставит реакцию 👀 вместо дубля.
11. **Параллельная сборка.** Пока Chromium рендерит, `metadata.fetch` тянет метаданные через httpx — две дорожки идут одновременно, результат сливается `merge_meta` (берётся самый полный заголовок/описание, цена/рейтинг из браузера приоритетнее).
12. **Карточка.** `build_message` собирает текст с Telegram-entities.
    - *entity* — разметка диапазона текста (offset/length): `code` для заголовка, `bold` для цены, `blockquote` для дисклеймера. Offsets считаются в единицах UTF-16 (`_utf16_len`) — именно так их считает Telegram, иначе эмодзи «съедут» разметку.
    - `_sender_prefix` дописывает автора в *тело* карточки: есть `@username` — обычный кликабельный текст; нет — `text_mention` (*упоминание без username* — единственный способ дать кликабельную ссылку на пользователя без @-имени). Так атрибуция переживает удаление исходного сообщения.
    - `trim_caption` режет подпись до лимита Telegram (1024 символа), не ломая entities.
13. **Отправка.** Один скриншот → `reply_photo`; несколько частей → `reply_media_group` (альбом). Telegram возвращает `file_id` (*идентификатор уже загруженного в Telegram файла* — повторная отправка по нему бесплатна и мгновенна) → он кладётся в кеш.

### 3.4 `queue_manager.py` — очередь и лимиты ресурсов

Один воркер (*worker* — фоновая задача, обрабатывающая очередь по одной), что согласовано с `SEMAPHORE = 1`. `enqueue` пропускает задачу через четыре фильтра по порядку:

1. **Дедупликация:** та же `(chat, топик, url)` уже выполняется → новый запрос подцепляется к существующему future (*future* — «обещание результата»: объект, который позже наполнится значением), без второго рендера.
2. **RAM-watchdog:** *сторож памяти* — если текущий RSS (*Resident Set Size* — реально занятая процессом RAM) выше `RAM_LIMIT_MB`, новые задачи отклоняются (анти-OOM).
3. **Per-chat квота:** один чат не может держать больше `MAX_INFLIGHT_PER_CHAT = 2` задач одновременно — один болтливый чат не забивает очередь всем (анти-DoS).
4. **Глубина очереди:** больше `MAX_QUEUE_SIZE` → `QueueFull`.

*DoS* (Denial of Service — отказ в обслуживании: атака, цель которой не украсть, а «положить» сервис). Воркер берёт задачу, прогоняет через `asyncio.wait_for` с таймаутом `TASK_TIMEOUT_SEC`; зависший сайт не держит очередь вечно. В `finally` всегда чистятся `_inflight` и per-chat-счётчик — утечки слотов нет даже при исключении.

### 3.5 `screenshot.py` — рендер в Chromium и нарезка

*Playwright* — библиотека управления настоящим браузером из кода. *Headless* — браузер без видимого окна, в памяти сервера.

- **Запуск (`_launch_args`).** Отключены GPU, расширения, фоновые сети, удалённые шрифты, WebRtc (*WebRTC* умеет выдать локальный IP контейнера — это leak инфраструктуры). Флаг `--no-sandbox` добавляется **только** если `CHROMIUM_SANDBOX` ≠ `on`.
  - *Sandbox* — «песочница»: ядро ОС изолирует процесс рендера так, что взлом страницы не даёт доступа к остальной системе. На Render Free песочница недоступна (нет user namespaces/seccomp), поэтому там `--no-sandbox`, но это компенсировано тем, что контейнер работает от **non-root** (`pwuser`): даже пробой рендера = непривилегированный пользователь, а не root. Локально/на VPS можно `CHROMIUM_SANDBOX=on` и получить настоящую изоляцию.
- **Stealth.** `_STEALTH_JS` — маленький init-скрипт (*init script* — JS, исполняемый до загрузки страницы), прячущий очевидный признак автоматизации (`navigator.webdriver`), чтобы сайты с анти-бот-защитой (Cloudflare и т.п.) не блокировали рендер.
- **Route handler — SSRF на каждый запрос.** `_route_handler` перехватывает *каждый* сетевой запрос страницы (подресурсы, iframe, fetch, редиректы) и проверяет `security.is_safe`. Это закрывает выход во внутреннюю сеть *изнутри* Chromium — то, что top-level-проверка пропустила бы. Заодно режутся медиа, websocket, шрифты, видео и рекламные домены (`AD_HOSTS`) — быстрее и легче по памяти.
- **Контекст.** `new_context` создаётся заново на каждый запрос (*context* — изолированный «профиль» браузера: свои cookie/хранилище) с `accept_downloads=False` (не сохраняем файлы со страницы) и `service_workers="block"` (*service worker* — фоновый скрипт, переживающий закрытие вкладки; блокируем, чтобы он не слал запросы к серверу атакующего).
- **Клэмп высоты — анти-OOM.** *Clamp* — жёсткое ограничение сверху. Высота захвата ограничивается `MAX_CAPTURE_HEIGHT = 2560` CSS-px **на уровне браузера**, а не пост-фактум в Pillow: `full_page` заставил бы Chromium отрисовать всю гигантскую страницу (листинги) в один bitmap до возврата байтов → взрыв памяти. Снимаем строго ограниченный viewport.
- **Нарезка (`_split_image`).** Высокий скриншот режется через Pillow на части по `PART_HEIGHT = 1280` px (Telegram-лимит ~10 МБ на фото), максимум `MAX_PARTS = 4`.
- **Рестарт браузера.** Каждые `RESTART_EVERY = 50` запросов браузер пересоздаётся — Playwright постепенно раздувает V8 heap (*heap* — куча, динамическая память; *V8* — движок JavaScript внутри Chromium); рестарт сбрасывает накопленное. Делается внутри семафора, без параллельных запросов.

### 3.6 `metadata.py` — быстрые метаданные без браузера

Тянет HTML напрямую через httpx (*httpx* — асинхронный HTTP-клиент) — это дешевле и быстрее браузера для текстовых данных.

- **Анти-OOM и анти-SSRF при скачивании.** Тело ответа читается потоково (*streaming* — по кускам, а не всё в память) с лимитом `MAX_BODY_BYTES = 2 МБ` — защита от gzip-bomb (*gzip-bomb* — крошечный архив, раздувающийся в гигабайты). Финальный URL после редиректов проверяется `is_safe` **до** чтения тела — не тянем внутреннюю страницу.
- **Парсинг.** Из HTML вытягиваются Open Graph (*Open Graph* — стандарт мета-тегов `og:*` для превью ссылок), Twitter Cards и обычные meta; заголовок, описание, картинка, имя сайта.
- **JSON-LD и `@graph`.** *JSON-LD* — структурированные данные о товаре (`@type: Product`) в `<script>`. `_walk_jsonld` рекурсивно обходит и контейнер `@graph`, где маркетплейсы (Rozetka/Comfy/Elmir) прячут товар не первым, — извлекаются цена, бренд, рейтинг.
- Несколько User-Agent перебираются по очереди (Slackbot первым — Cloudflare пропускает ботов соцсетей); берётся первый удачный ответ. Сырые URL/исключения в логи не пишутся (анти-утечка).

### 3.7 `cache.py` — кеш в памяти

Хранит только `file_id` + метаданные (не байты картинок), ничего на диск.

- **Ключ.** `_canonical` нормализует URL (нижний регистр схемы/хоста, срез фрагмента `#...`; query `?id=...` сохраняется — для товарных страниц он значим), затем `sha256` (*хеш-функция*, даёт фиксированный идентификатор).
- **Дифференцированный TTL.** *TTL* (Time To Live — срок жизни записи). Разный по типу контента: фото 3600 сек, страница с ценой — 900 (цена устаревает быстрее), текст без скриншота — 300, негативный кеш ошибки — 180.
- **Негативный кеш.** `save_failure` коротко (180 сек) помнит, что страница не открылась, — чтобы не долбить заведомо мёртвый сайт повторно.
- `TTLCache(maxsize=200)` сам вытесняет старое и не растёт.

### 3.8 `main.py` — сборка, health и graceful shutdown

- **Логирование.** `loguru` настроен с `backtrace=False, diagnose=False` — чтобы в трейсы не попадали значения переменных (включая токен/URL). Уровень — из `LOG_LEVEL`.
- **HTTP-эндпоинты.** `/` и `/ping` — для healthcheck/keep-alive. `/health` отдаёт только статус и флаги (`browser`, `bot`) — без внутренних счётчиков очереди/кеша (чтобы не давать разведданные для DoS).
  - *Health endpoint* — точка, по которой хостинг/монитор проверяет, жив ли сервис.
- **Запуск.** Инициализируется браузер, регистрируется процессор очереди и воркер, поднимается aiogram polling и uvicorn-сервер в одном `asyncio.gather`.
  - *Long-polling* — бот сам периодически спрашивает Telegram «есть новое?», держа исходящее соединение (в отличие от webhook, где Telegram стучится к боту).
- **Graceful shutdown.** *Корректное завершение*: по `SIGTERM` (сигнал «заканчивай») бот останавливает polling, ждёт освобождения семафора (до 60 сек — чтобы дорисовать текущий скриншот), аккуратно закрывает браузер и Playwright. Сигнал ловится через `get_running_loop` (актуальный API вместо устаревшего `get_event_loop`).

---

## 4. Жизненный цикл одного запроса (сквозной таймлайн)

1. Юзер кидает ссылку → `handle()` проверяет возраст, группу, топик, длину URL.
2. Не админ, не дубль, не слишком часто → `is_safe(url)` подтверждает безопасность.
3. `cache.get` пуст → задача в очередь (прошла RAM-watchdog и per-chat-квоту).
4. Бот отвечает плашкой «⏳ готую безпечний перегляд» с позицией в очереди.
5. Воркер запускает `screenshot.shoot` (повторный `is_safe`, новый изолированный контекст, route handler фильтрует подресурсы), параллельно — `metadata.fetch`.
6. Chromium рендерит, Pillow режет; метаданные сливаются.
7. Карточка (превью + название/цена/рейтинг + дисклеймер + автор) уходит в чат; `file_id` сохраняется в кеш; плашка удаляется.
8. Следующий такой же запрос в течение TTL → мгновенный ответ из кеша.

## 5. Безопасность — что закрыто и почему

| Угроза | Расшифровка | Как закрыто |
|---|---|---|
| SSRF / internal access | сервер шлют во внутреннюю сеть | `is_safe` на 4 рубежах, резолв всех A+AAAA, блок-диапазоны, порты 80/443 |
| DNS rebinding | домен «перепрыгивает» с публичного IP на внутренний | повторный `is_safe` перед `goto` + проверка каждого подзапроса |
| Cloud metadata | кража облачных ключей с `169.254.169.254` | адрес в блок-листе link-local |
| Container RCE | выполнение чужого кода в контейнере | свежий Chromium + non-root `pwuser` (+ sandbox локально) |
| DoS / OOM | «положить» сервис, исчерпать память | per-chat-квота, RAM-watchdog, лимит тела httpx, clamp высоты |
| Open proxy | бот работает где попало | fail-closed allow-list групп |
| Secret leak | утечка токена в логи | `loguru` без diagnose; логируем тип ошибки, не сырьё |

*RCE* (Remote Code Execution — удалённое выполнение кода: атакующий запускает свой код на твоём сервере) — самый тяжёлый класс; именно поэтому браузер свежий и не от root.

## 7. Деплой на Render (Free)

1. Залить код в новый GitHub-репозиторий.
2. Render → **New +** → **Blueprint** → выбрать репозиторий (подхватит `render.yaml`).
3. В Dashboard ввести секреты: `BOT_TOKEN`, `ALLOWED_GROUP_IDS`.
4. После деплоя в логах должны появиться `Browser started` и `Queue worker started`.

**Ограничения Render Free (честно):**

- **Засыпание.** Free web-сервис останавливается без входящего HTTP. Бот на long-polling его не удерживает. Бесплатное решение — внешний пингер (UptimeRobot / cron-job.org) на `/ping` раз в ~10 мин.
- **`--no-sandbox` неизбежен** — нет user namespaces/seccomp. Компенсация: non-root + свежий Chromium. Полная изоляция — локально или на VPS.
- **Egress-фильтра нет** (*egress filter* — фильтр исходящего трафика на уровне сети). На Free защита от SSRF держится на app-уровне (`is_safe`). Для жёстких сценариев — VPS с `nftables`.

## 11. Глоссарий (быстрый справочник)

- **SSRF** — заставить сервер сходить туда, куда атакующий сам не может (внутренняя сеть, метаданные облака).
- **DNS rebinding** — домен резолвится сперва в публичный, потом во внутренний IP, обходя одноразовую проверку.
- **Cloud metadata** — внутренний адрес `169.254.169.254`, отдающий облачные ключи; цель SSRF.
- **fail-closed / secure-by-default** — при неопределённости запрещать, а не разрешать.
- **allow-list / denylist** — список разрешённого / запрещённого.
- **headless / sandbox** — браузер без окна / изоляция процесса рендера ядром ОС.
- **route handler** — перехватчик каждого сетевого запроса страницы.
- **OOM / RSS / RAM-watchdog** — нехватка памяти / занятая память / сторож, режущий новые задачи у лимита.
- **DoS** — атака «положить сервис».
- **rate limit / dedup / in-flight** — ограничение частоты / отсев повторов / задача в работе прямо сейчас.
- **TTLCache / TTL / file_id** — самоистекающий кеш / срок жизни записи / id уже загруженного в Telegram файла.
- **entity / UTF-16 offset** — разметка диапазона текста в Telegram, считаемая в единицах UTF-16.
- **future / worker / semaphore** — обещание результата / фоновый обработчик / счётчик одновременного доступа.
- **long-polling vs webhook** — бот сам спрашивает Telegram / Telegram стучится к боту.
- **graceful shutdown / SIGTERM** — корректное завершение по сигналу «заканчивай».
- **RCE** — выполнение чужого кода на твоём сервере.
- **non-root / capabilities / seccomp / egress filter** — работа не от админа / точечные привилегии / фильтр системных вызовов / фильтр исходящего трафика.
- **JSON-LD / Open Graph / @graph** — структурированные данные о товаре / мета-теги превью / контейнер, где маркетплейсы прячут товар.
